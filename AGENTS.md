# Knovana Agent 操作指南

本文档用于指导后续 agent 在本仓库中的代码、文档、测试和提交前检查工作。目标是让每次变更都更容易被人理解、维护和扩展。

## 1. 基础环境约定

- 当前开发环境是 **Windows + PowerShell**，不要默认按 Linux shell 习惯操作。
- 读取中文文档和源码时必须明确使用 UTF-8 编码，避免 PowerShell 默认编码导致乱码。
- PowerShell 读取文本优先使用：

```powershell
Get-Content -Encoding UTF8 path\to\file.md
Select-String -Path path\to\file.md -Pattern "keyword" -Encoding UTF8
```

- 搜索文件和内容优先使用 `rg`。`rg` 对 UTF-8 文本支持良好，也比递归 `grep`/`Get-ChildItem` 更快。
- 路径书写优先使用 Windows 路径或 PowerShell 兼容写法，例如 `docs\backend-service-design.md`。
- 不要假设 WSL、bash、Linux 工具链可用。只有在项目明确要求或用户明确授权时才使用 WSL/bash。

## 2. 代码质量原则

- 代码首先是给人读的，其次才是给机器执行的。实现时优先考虑清晰性、可维护性和可扩展性。
- 遵循现有项目结构和命名风格，不为了展示技巧引入不必要的抽象。
- 把复杂度放在合适的位置：
  - API 层保持薄，只处理路由、认证、校验和响应格式。
  - Service 层承载业务编排，避免依赖具体 HTTP Context。
  - Storage 层隔离数据库和文件系统细节。
  - Agent 层只处理需要 AI 能力的流程。
- 新增功能时优先设计可测试的函数、类和模块边界。
- 不要为了局部方便牺牲长期一致性，例如绕过统一错误格式、跳过 Schema 校验、直接操作跨层依赖。

## 3. 重要功能变更流程

- 涉及较重要功能更改时，必须先形成推荐方案，等待用户确认后再调整代码。
- 推荐方案应包含：
  - 目标和非目标，避免范围失控。
  - 影响范围，包括前端、后端、数据模型、API、文档、测试和部署。
  - 关键技术选择和取舍理由。
  - 兼容性、迁移成本和主要风险。
  - 建议的实施步骤和验证方式。
- 用户确认前可以阅读代码、梳理现状、提出方案和风险，但不要直接修改实现代码。
- 如果只是小范围 bugfix、文案调整、配置修正或用户已明确要求直接修改，可以按常规流程执行。

## 4. 注释原则

- 避免不必要的注释。不要解释显而易见的语句，例如“设置变量”“调用函数”。
- 关键决策必须注释，方便后期维护者理解为什么这样做。适合注释的内容包括：
  - 重要的业务规则和边界条件。
  - 安全相关判断，例如权限、路径校验、输入过滤。
  - 非直觉的兼容性处理，例如浏览器扩展、SSE、Windows 路径或编码问题。
  - 与外部 SDK、Agent 工具或协议相关的约束。
- 注释要短，说明原因或约束，不复述代码表面行为。

## 5. 测试要求

- 增加或调整功能后必须考虑测试，覆盖范围要和风险匹配。
- 增加关键功能后必须编写或更新测试用例。不能只做手动验证，也不能只描述“应该测试”。
- 前端变更至少关注：
  - 组件行为和状态变化。
  - 关键用户流程。
  - 与后端 API 的请求、响应和错误处理。
  - 必要时做浏览器端手动验证或截图验证。
- 后端变更至少关注：
  - Service 层单元测试。
  - API 路由集成测试。
  - 权限、输入校验、错误格式和 SSE 流式输出。
  - Storage 和 Agent 工具的边界条件。
- 如果暂时无法运行测试，要在最终说明中明确原因和剩余风险。
- 不要只运行与变更无关的测试来替代真正相关的验证。

### 5.1 前端质量验证命令

当前前端位于 `chrome-extension/`。前端代码变更后，在该目录下优先按以下顺序验证：

```powershell
pnpm test:unit              # 1) 单元测试通过
pnpm format                 # 2) 自动格式化，会写入修改
pnpm lint                   # 3) lint 通过，包含 prettier --check + eslint
pnpm check                  # 4) svelte-check 类型检查，要求 0 error
```

- 如果 `pnpm format` 写入了修改，必须重新检查 `git diff`，必要时再次运行相关测试。
- 如果新增关键 UI 交互、状态逻辑、API 客户端行为或数据转换逻辑，应补充对应单元测试。
- 如果变更影响真实用户流程，应在单元测试之外补充必要的手动验证说明。

### 5.2 后端质量验证命令

后端计划采用 Node.js + TypeScript + Hono。后端目录创建后，`backend/package.json` 必须维护并使用明确的质量验证脚本：

```powershell
pnpm test                  # 后端测试通过
pnpm format                # 自动格式化，会写入修改
pnpm lint                  # lint 通过
pnpm check                 # TypeScript 类型检查通过
```

- 后端关键功能必须补充测试用例，尤其是 Service、API 路由、权限、输入校验、错误格式、SSE 流式输出和 Storage/Agent Tool 边界条件。
- 如果后端脚本名称与上面不同，必须以 `backend/package.json` 的实际脚本为准，并在最终说明中写清楚实际运行命令。

## 6. 文档同步

- 新增或调整功能时，如果影响架构、API、数据模型、运行方式、配置、部署或用户流程，必须同步更新相关文档。
- 文档和代码实现要尽量保持一致，尤其是：
  - `docs/architecture-overview.md`
  - `docs/backend-service-design.md`
  - `docs/api-specification.md`
  - `docs/data-model.md`
  - `docs/chrome-extension-design.md`
- 更新文档时要写当前真实实现或明确的目标方案，不保留已经废弃的实现描述。
- 代码示例、文件路径、依赖名、启动命令和测试命令必须和当前技术栈一致。
- 文档里涉及中文内容时，读取和检查必须显式使用 UTF-8 编码。

## 7. 格式化和一致性

- 代码提交前必须 format。优先使用项目已有格式化命令，不要手工制造风格差异。
- 只格式化本次变更相关文件，除非用户明确要求全仓库格式化。
- 如果项目已有 lint、typecheck/check、format、test 脚本，应优先使用这些脚本。
- 前端默认使用第 5.1 节的验证命令；后端默认使用第 5.2 节的验证命令。
- 格式化命令写入修改后，必须重新查看 diff，确认没有引入无关变更。

## 8. Git 提交前安全检查

提交或建议提交前，至少完成以下检查：

- 查看工作区状态，确认只包含本次任务相关变更：

```powershell
git status --short
git diff --stat
git diff
```

- 不要回退、覆盖或删除用户已有变更，除非用户明确要求。
- 检查是否误加入敏感信息：
  - API key、token、JWT secret、cookie、私钥、`.env` 实际配置。
  - 本地绝对路径、个人账号信息、生产域名密钥。
  - 大文件、构建产物、缓存目录、`node_modules/`。
- 建议使用搜索做基础敏感信息检查：

```powershell
rg -n "api[_-]?key|secret|token|password|PRIVATE KEY|BEGIN RSA|ANTHROPIC_API_KEY|JWT_SECRET" -S .
```

- 运行与变更相关的 format、lint、typecheck 和 test。
- 复查文档是否与代码一致，尤其是 API、配置、路径和命令。
- 提交信息应简洁描述用户可理解的变更，不夸大范围。

## 9. 交付说明

- 最终回复要说明改了什么、验证了什么、还有什么未验证。
- 如果涉及文件，优先给出具体文件路径。
- 如果未能运行测试或检查，要明确说明原因。
- 不要把内部思考、无关日志或过长 diff 塞给用户；保留高信号结论。

> 本规约会随项目演进而更新，每次更新需在对话中明确告知用户并征得同意。
